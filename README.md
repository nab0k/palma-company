## WHERE FILES LIVE / ГДЕ ЖИВУТ ФАЙЛЫ

GitHub хранит каноническое закоммиченное состояние Palma. Локальная папка
репозитория — общая рабочая копия: Claude и Codex ищут и изменяют файлы здесь,
каждый в пределах назначенной задачи. ChatGPT читает закоммиченные файлы через
GitHub. Незакоммиченный локальный файл не считается переданным ChatGPT.

Книга и редакционная система находятся в `palma-method/`. Сайт компании и
другие рабочие области остаются в своих существующих путях. Notion показывает
статусы, ссылки и dashboard, но не хранит отдельную каноническую версию текста.
Синхронизация разрешена только в направлении GitHub → Notion.

Перед началом работы агент сначала ищет входные файлы в репозитории, проверяет
состояние Git и обновляет сведения о GitHub. Если рабочая папка чистая и ветку
можно обновить без объединения разных версий, используется только безопасное
fast-forward обновление. При незакоммиченных изменениях или расхождении веток
агент останавливается и сохраняет существующую работу.

После работы агент проверяет изменения, коммитит только файлы своей задачи,
отправляет отдельную ветку и открывает pull request. Для передачи работы другому
агенту достаточно сообщить репозиторий, ветку, commit SHA и пути файлов.

### Ежедневный workflow

1. Открыть локальный репозиторий.
2. Проверить `git status`.
3. Получить сведения об изменениях с GitHub и безопасно обновить чистую ветку.
4. Создать или открыть рабочую ветку задачи.
5. Выполнить только разрешённую задачу.
6. Проверить изменённые файлы и запустить применимые проверки.
7. Создать commit.
8. Отправить ветку на GitHub.
9. Открыть pull request.
10. Передать другим агентам только имя репозитория, ветку, commit SHA и пути.

Чтобы избежать конфликтов, одна задача должна иметь одну рабочую ветку и
точные входные и выходные пути. Не следует одновременно редактировать один файл
в разных ветках и создавать копии с названиями `final`, `new` или `latest`.

После ошибки не удаляйте файлы и не переписывайте историю. Сначала остановите
работу, сохраните `git status`, имя ветки и commit SHA, затем восстановите файл
из известного коммита или откройте отдельную исправляющую задачу. Принятые
решения и версии документов остаются прослеживаемыми.
